# 📜 全社絶対ルール (Global Rules)

> このファイルはすべての役職が最初に読み込む絶対不可侵のルールである。
> いかなる場合でも、ここに記載されたルールに違反してはならない。

---

## 1. 【管轄絶対】 Action Scope の厳守

自身の管轄外のファイルやディレクトリは **絶対に直接編集してはならない**。
他領域の情報を知る必要がある場合は、必ず `docs/` 内のドキュメントを参照すること。

### 管轄領域一覧

| 役職 | 管轄領域 |
|---|---|
| Secretary / CEO | `.company/board.md`, `.company/phase.md` |
| PM | `docs/prd/` |
| Architect | `docs/api-specs/`, `.company/adr/` |
| Frontend Engineer | `src/frontend/` |
| Backend Engineer | `src/backend/` |
| AI Engineer | `src/ai/`, `docs/prompts/` |
| DevOps Engineer | `infra/`, `.github/` |
| QA Engineer | `tests/`（プロジェクト全体の**読取権限**あり） |

---

## 2. 【ピアレビュー厳守】 QA ゲートの必須通過

実装後、直接社長へ報告することは **禁止**。
必ず **QA Engineer** のテストとレビューを通すこと。

### フロー
```
実装完了 → QA Engineer がテスト・レビュー → パス → 社長へ報告
                                          → 不合格 → 該当エンジニアへ差し戻し
```

---

## 3. 【ADR記録】 技術決定の永続化

重要な技術スタックや設計方針の決定は、後から背景を辿れるよう必ず `.company/adr/` に理由と共に残すこと。

### ADR テンプレート
```markdown
# ADR-XXXX: [タイトル]
- **ステータス**: 提案 / 承認済み / 却下
- **日付**: YYYY-MM-DD
- **決定者**: [役職名]

## コンテキスト
[なぜこの決定が必要だったか]

## 決定
[何を選択したか]

## 理由
[なぜその選択をしたか、代替案との比較]

## 影響
[この決定によって生じる影響]
```

---

## 4. 【遅延読み込み】 コンテキスト管理

- タスク着手前に `.company/roles/index.md` を確認し、適切な役職を特定する
- 現在のタスクに必要な **単一の役職ファイルのみ** を読み込む
- 他の役職ファイルは意図的に読み込まない
- 完全にその役職のスコープとスキルセットに没入して作業を行う

---

## 5. 【承認ゲート】 フェーズ遷移の承認

各フェーズの完了時には、必ず **社長（ユーザー）の承認** を得てから次のフェーズに進むこと。
承認なしにフェーズを進行させることは禁止する。

---

## 6. 【ドキュメント駆動】 情報共有の原則

- 部署間の情報共有は `docs/` 配下のドキュメントを介して行う
- 直接的な部署間通信（ロールファイルの同時読み込み等）は禁止
- すべての成果物はドキュメントとして永続化すること
